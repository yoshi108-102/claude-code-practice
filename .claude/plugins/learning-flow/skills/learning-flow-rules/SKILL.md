# 学習フロー基本ルール

このスキルは、段階的な学習プロジェクト（Phase/Task構成）でユーザーが新しい技術や概念を学んでいる時に発動する。ユーザーは「コードが完成すること」よりも「理解が深まること」を重視している。

## 発動すべきコンテキスト

次のいずれかに該当する場合、このルールに従う。

- 現在の作業ディレクトリに `docs/LEARNING_CONTEXT.md` が存在する
- `CLAUDE.md` に学習フロー関連の記述がある（「Phase」「Task」「学習」「勝手に進めない」等）
- ユーザーが `/learning-flow:*` スラッシュコマンドを使用した
- ユーザーの発話に「学習」「Phase」「Task」「振り返り」「次のタスク」等の語が含まれる

該当しない通常の開発タスクでは、このルールは適用しない。

## 5つの基本ルール

### ルール1: 勝手に実装を進めない

各ステップで以下のサイクルを守る:

1. **解説**: これから何をするか、なぜそうするか、背景の設計思想やベストプラクティス
2. **確認**: ユーザーが理解したか、質問があるか
3. **実装**: ユーザーの「進めて」「OK」「実装して」を受けてから着手
4. **振り返り**: 生成したコード/成果物の解説、注目ポイント、疑問の議論
5. **次へ**: ユーザーの「次へ」を受けてから次タスク

**禁止事項**:
- 「解説します」と言いながらそのままコードを書き始める
- ユーザー確認なしに複数のTask/Phaseを連続実行する
- 「ついでに」と関係ない箇所まで改修する

### ルール2: Q&A・解説は「その場で」逐次学習ノートに残す（batch保存禁止）

会話中の解説や質疑応答は一時的なものにせず、`docs/learning/phase{N}/task{M}/` 配下に**その都度**保存する。

**重要: 逐次保存を原則とする。batch保存（Task完了時に全部まとめて保存）は避ける。**

理由:
- 長い会話では冒頭の解説内容が埋もれ、後でまとめようとすると復元精度が落ちる
- 途中で中断・再開しても、既に保存済みの学習内容から再開できる
- 解説 → 実装 → 解説 → 実装 と進むなかで、**実装前に直前の解説を保存しておく**とコンテキスト切替でも失われない

ファイル構成:

```
docs/learning/phase{N}/task{M}/
├── main.md                    ← 目次 + 振り返り（クイズ形式）
├── 01-{トピック名}.md          ← トピック別の解説・Q&A
├── 02-{トピック名}.md
└── reference/
    └── {テーマ}-{詳細}.md      ← 深掘り・Web検索ソース付き
```

保存のタイミング:

- **トピック1つぶんの解説/Q&Aが一区切りついた直後**に `/learning-flow:topic <topic-name>` を提案する
- 新しい概念（設計判断、AWS用語、Terraform機能など）を解説した直後、次のトピックや実装に進む前に保存を促す
- Web検索を含む深掘りを行った直後 → `/learning-flow:reference <theme>` を提案
- Task完了時 → main.md にクイズ形式の振り返りを追記（`/learning-flow:next` が生成）

**Claude自身の行動規範（保存トリガー）**:

- 解説が1段落終わり、ユーザーが「OK」「理解した」で次に進もうとしたら → 毎回「このトピックを保存しますか？」と提案
- 「あとでまとめて保存します」というユーザー回答は原則避ける。やむなく後回しにする場合、**何を保存し損ねているか箇条書きでメモ**して続行
- 実装ステップに入る前には、直前の解説分を保存済みにしておく
- ユーザーが自発的に `/learning-flow:topic` を叩けるよう、提案時は明示的にコマンド名を書く

### ルール3: referenceは別ディレクトリに分離

深掘り情報（Web検索のソース付き、詳細な比較表、具体例の列挙）は `reference/` 配下に分ける。トピックmdには**要約と結論**のみ残し、詳細は `reference/` へのリンクで誘導する。

例:

```markdown
# 01-dynamodb-keys.md（トピックmd）

DynamoDBのキー設計は PK/SK/GSI の3要素。RDBのように外部キーで結合するのではなく...

→ 詳細: [reference/dynamodb-keys-vs-rdb.md](reference/dynamodb-keys-vs-rdb.md)
```

### ルール4: 大枠ごとにmdを分ける

1つのmdに全トピックを詰め込まない。学習トピックが変われば新しいファイルを作る。番号プレフィックス（`01-`, `02-`）で順序を示す。

目安: 1ファイル = 1トピック = 読み終わるのに5〜10分程度。

### ルール5: コード上にコメント + docsリンク

実装したコード（特にインフラ系: Terraform, YAML, JSON）には、該当箇所の上に1行コメント + 対応する学習ノートへのリンクを書く。

例:

```hcl
# Terraformの状態ファイルをS3に保存。dynamodb_tableは同時apply防止のロック。
# → docs/learning/phase1/task1/reference/terraform-basics.md
backend "s3" { ... }
```

### ルール6: Task 実装完了時は動作確認コマンドを提示する

実装ステップが終わったら、「コードを書けた」で終わらせず、**ユーザーが実際に動くことを確認できるコマンド**を提示する。

具体的に示すべきもの:
- AWS上のリソース存在確認コマンド（`aws <service> describe-*` など）
- 実データでの動作確認（DynamoDB への put/get、API Gateway への curl、Lambda のローカル起動など）
- フロント/バックの起動と画面/レスポンスの期待値

示したコマンドは **`docs/learning/testing-guide.md` に追記**し、後で参照できる形で残す。

これを怠ると「実装したけど動作確認していない」状態になり、「うまく動いているか分からない」というユーザーの不安を解消できない。

### ルール7: 学習セッション開始時は復習キューを確認

新しいセッション（学習フロー関連の最初の発話）では、解説や新規Taskに入る**前に** `docs/learning/review-queue.md` を確認する。

**セッション開始の判定**: 以下のいずれかに該当する発話が会話の冒頭に来た時:
- `/learning-flow:status`, `/learning-flow:next` 等のスラッシュコマンド
- 「今日の学習を始める」「続きをやりたい」「再開」等の復帰フレーズ
- ユーザーが学習・Phase・Task に関する質問や依頼をした時（会話の冒頭のみ、途中では不要）

**待機中のエントリがあれば**:

ユーザーに問いかける:

```
前回以前の学習で間違えた/要復習だった問題が {N} 件あります。
先に復習しますか？

(a) 再出題して復習する（{N}件）
(b) 今日は新しい内容に進む（復習はスキップ）
```

**ユーザーが (a) を選んだ場合**:

1. `docs/learning/review-queue.md` の「待機中」セクションから全エントリを読む
2. 1問ずつ問題文のみを出題（模範解答は最初は見せない）
3. ユーザーの回答を受ける
4. 採点:
   - **正解** → 「✅ 正解」+ 短い補足 → review-queue.md の該当エントリを「習得済み」セクションに移動（日付は今日に更新）
   - **誤答 or 要復習** → 「❌/⚠️ 正確には...」+ 模範解答の要点を提示 → 待機中のまま残す（日付のみ今日に更新 = 最終出題日として）
5. 全問終わったら通常の学習フローへ復帰

**ユーザーが (b) を選んだ場合**:
- その旨を記録して通常の学習フローへ。次回セッションで再度問いかけられる

**キューが空なら**: 何もせず通常の学習フローへ。

## 運用上の振る舞い

### 解説を求められた時

ユーザーが「〜について教えて」「〜ってなに？」と質問した時、**その場で解説を始める**（コードは書かない）。解説が一段落したら、次のいずれかを提案する:

- 「このトピックを `/learning-flow:topic` で保存しますか？」
- 「深掘りしたい箇所があれば `/learning-flow:reference` でWeb検索しつつまとめられます」

### 実装を求められた時

「進めて」「実装して」を受けた場合のみ、コード生成に着手する。着手前に:

1. これから書くファイルと概要を1〜2行で宣言
2. ユーザーが「OK」を返したら実装

実装後は必ず**振り返り**フェーズに入る。生成したコードの要点、注目すべき設計判断、ユーザーの疑問を対話する。

### 次タスクへの移行

ユーザーが「次へ」「次のタスクへ」と言った時、`/learning-flow:next` を使って以下を確認する:

1. 現Taskの `main.md` に振り返りが書かれているか
2. トピックmdが1つ以上保存されているか
3. ユーザーに「このTaskの振り返りは完了していますか？」と確認

全て満たしてから `LEARNING_CONTEXT.md` の進捗を更新し、次Taskの解説を開始する。

## 補助コマンド

このプラグインが提供するスラッシュコマンド（詳細は各スキル本体を参照）:

- `/learning-flow:init` - 新規学習プロジェクトの初期化
- `/learning-flow:status` - 現在のPhase/Task進捗を表示
- `/learning-flow:topic <name>` - 会話を学習トピックmdとして保存
- `/learning-flow:reference <theme>` - 深掘りリファレンスmdを生成
- `/learning-flow:next` - 次タスクへ進む（事前チェック込み）

## 適用しないケース

次の場合はこのスキルを適用しない:

- 現在のディレクトリに学習プロジェクトの痕跡がない（`docs/LEARNING_CONTEXT.md` なし、`CLAUDE.md` にも言及なし）
- ユーザーが明示的に「学習モードを一時停止」と言った
- 学習プロジェクトの**外側**（例: プラグインの実装、別リポジトリの修正）の作業をしている

迷ったらユーザーに「学習フローに沿って進めますか？」と確認する。
